### 5. 飞轮架构重设计

提出并验证了三层飞轮模型：

```
第一层：项目内 Harness 进化（需求 → 执行 → 观察失败 → 改进 CLAUDE.md）
第二层：跨项目模式迁移（项目 A 经验 → Skill → 项目 B 受益）
第三层：Harness 改进 Harness（用 Claude Code 写 Skill → Skill 让 Claude 更高效 → 写出更好的 Skill）
```

**核心发现**：飞轮的引擎是"观察失败 → 改进"这一步。旧版设计需要用户手动触发 /harness-review，飞轮容易断裂。

**重设计方案**：引入 `.harness/` 共享数据中心，让 Skill 自动联动：

```
.harness/
├── config.json          ← harness-init 写入（技术栈、命令）
├── phase-journal.jsonl  ← harness-resume 每 Phase 自动写入（执行数据）
└── review-log.md        ← harness-review 写入（改进记录）→ harness-plan 读取（避坑）
```

关键改动：**harness-resume 每个 Phase 完成后自动执行轻量级 review + 写入飞轮日志**，不再依赖用户手动推动。

---

## 关键洞察

1. **飞轮的前提条件**：每一轮都要有"观察失败 → 改进"步骤，否则飞轮不转
2. **harness-review 是飞轮引擎，不是可选项**
3. **自动化 > 手动触发**：将 review 嵌入 resume 比依赖用户记忆更可靠
4. **Memory 的三层隔离**：全局 / 项目级 / CLAUDE.md，按启动目录自动隔离
